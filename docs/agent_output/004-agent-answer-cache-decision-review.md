# Agent 回答缓存决策复盘

## 背景

Phase 10 曾考虑通过 Redis 缓存 Agent 最终回答来降低 token 成本。初始设想是只缓存“明确查询类问题”的最终回答，并通过查询指纹、用户权限、日期、模型名、prompt 版本等字段构造缓存 key。

经过重新评估后，该方案决定放弃。Phase 10 不再实现 Redis 缓存、Agent 回答缓存、查询指纹缓存，也不再把缓存作为 token 成本控制手段。

## 放弃原因

### 1. Agent 回答缓存 key 离散度过高

Agent 最终回答不是固定函数输出，它同时依赖：

- 用户角色与数据权限范围。
- 用户所属大区或销售员身份。
- 查询日期范围。
- 查询指标、维度、排序方式和 TopN。
- 当前会话上下文。
- 模型名称与提示词版本。

这些因素组合后，缓存 key 类型会非常多。即使两个问题语义接近，只要权限范围、日期范围或查询范围不同，就不能共享缓存。

结果是：缓存系统复杂度提高，但命中率很可能很低。

### 2. 权限隔离会进一步压低命中率

销售分析 Agent 必须严格遵守权限隔离：

- 总监可看全公司。
- 主管只能看本大区。
- 销售员只能看本人或被允许的个人范围数据。

如果缓存 key 不包含权限范围，存在越权复用风险；如果缓存 key 包含权限范围，跨用户命中率会明显下降。

因此 Agent 回答缓存很难同时做到高命中率和安全隔离。

### 3. LLM 或 embedding 归一化会引入新的 token 成本

可以尝试用 LLM 分析语义，或用 embedding 做相似度匹配，以提升自然语言问题的缓存命中率。

但这会带来新的问题：

- 每次查询缓存前可能要额外调用 LLM 或 embedding。
- 命中率不高时，额外调用反而增加成本。
- 相似命中还可能带来错误复用或权限边界误判。

因此，用额外模型调用换缓存命中，在本项目当前阶段不划算。

### 4. Agent 最终回答不是最适合缓存的层级

如果目标是提高响应速度，缓存可以放在 Service 层，因为 Service 方法数量有限、参数结构稳定、输出确定性更强。

但如果目标是降低 token 成本，缓存 Agent 最终回答并不理想：

- Agent 输入是自然语言，表达空间大。
- Agent 输出受模型表达影响，不完全稳定。
- 缓存命中前仍需要复杂判断。

因此，本项目暂不把 Agent 回答缓存作为降本策略。

## 经验归纳

### 1. 缓存应优先放在确定性边界

适合缓存的位置通常是：

- 参数结构稳定。
- 输出确定性强。
- 权限范围清晰。
- 命中率可预估。

Service 层统计查询比 Agent 最终回答更符合这些条件。

### 2. Token 成本控制优先减少输入上下文

对当前项目来说，更可靠的降本方向是减少每轮请求携带的历史 token。

因此 Phase 10 收敛为：

- 使用 LangChain 官方 `SummarizationMiddleware` 做消息摘要压缩。
- 数据库只保存与大模型上下文一致的 `user/assistant` 消息。
- 不保存 `tool` 消息到长期记忆。
- 达到 20 条对话消息时触发摘要。
- 摘要后保留最近 6 条对话消息。

### 3. 不为低命中率缓存引入高复杂度

缓存不是天然收益。缓存方案至少要同时满足：

- 命中率足够高。
- 缓存 key 可控。
- 失效策略简单。
- 不破坏权限隔离。
- 不需要额外消耗大量 token 才能判断是否命中。

如果这些条件不成立，缓存会变成复杂度负债。

## 当前决策

Phase 10 删除以下内容：

- Redis 缓存策略。
- Agent 最终回答缓存。
- `QueryFingerprintService`。
- `AgentAnswerCacheService`。
- 缓存 key 设计。
- TTL 设计。
- 缓存命中/未命中/降级测试。

Phase 10 保留以下内容：

- 消息摘要压缩。
- `SummarizationMiddleware` 官方接入。
- 数据库会话消息与大模型上下文一致。
- `tool` 消息不进入长期记忆。
- Token 成本通过减少历史上下文长度来控制。
