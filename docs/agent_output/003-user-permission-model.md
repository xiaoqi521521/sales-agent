# 用户权限模型与异常预警可见范围

本文档说明当前智能销售 Agent 的用户认证、角色权限、数据隔离规则，以及不同角色在“异常预警”工具中能获取的数据差异。

## 一、认证身份来源

当前项目不单独维护 `user` 表，开发阶段直接复用销售员表 `sa_sales_rep` 作为登录身份来源。

登录接口：

```http
POST /auth/login
```

请求示例：

```json
{
  "repId": 2
}
```

登录成功后返回 JWT：

```json
{
  "accessToken": "...",
  "tokenType": "bearer",
  "user": {
    "repId": 2,
    "username": "张伟",
    "role": "SALES_REP",
    "regionId": 1
  }
}
```

后续调用受保护接口时，请求头需要携带：

```http
Authorization: Bearer <accessToken>
```

## 二、当前用户上下文

系统解析 JWT 后，会构造 `CurrentUser`，当前只保留权限判断需要的字段：

```text
username
role
region_id
rep_id
```

其中：

- `username`：当前登录用户名称。
- `role`：当前登录用户角色。
- `region_id`：用户所属大区。
- `rep_id`：用户对应销售员 ID。

## 三、角色定义

当前系统支持三种角色：

| 角色 | 中文含义 | 数据范围 |
| --- | --- | --- |
| `SALES_REP` | 普通销售员 | 只能查看自己的数据 |
| `SALES_MANAGER` | 销售主管 | 只能查看自己所属大区的数据 |
| `SALES_DIRECTOR` | 销售总监 | 可以查看全公司数据 |

## 四、权限过滤位置

权限过滤主要发生在 `SalesQueryService`，而不是只依赖 API 层。

这样设计的目的：

- API 层负责认证，确认“你是谁”。
- Agent 层负责对话编排，不直接访问数据库。
- Tool 层负责调用业务能力，不自行决定权限范围。
- Service 层负责业务查询和权限过滤，是数据隔离的核心边界。
- Repository 层只负责执行具体数据查询，不理解用户角色。

因此，Agent 即使调用工具，也只能通过 Service 获取当前用户权限范围内的数据。

## 五、常规销售数据可见范围

### 1. 普通销售员：`SALES_REP`

普通销售员只能看到自己的销售数据。

具体表现：

- 查询订单时，强制使用当前用户自己的 `rep_id`。
- 查询销售额时，只统计当前销售员自己的订单金额。
- 查询销售员排行时，不返回“只有自己一行”的假排行，而是提示当前账号无权查看团队销售员排行，并引导查询个人业绩。
- 查询大区排行时，不返回“自己金额挂到本大区”的假排行，而是提示当前账号无权查看大区排行。
- 查询产品排行和月度趋势时保留，但语义是“个人产品销售排行”和“个人销售趋势”，只基于自己的订单数据。
- 如果显式询问其他销售员或其他大区，系统不会返回越权数据。

### 2. 销售主管：`SALES_MANAGER`

销售主管只能看到自己所属大区的数据。

具体表现：

- 查询订单时，强制限定为当前用户的 `region_id`。
- 查询销售额时，只统计本大区金额。
- 查询销售员排行时，只包含本大区销售员。
- 查询大区排行时，只返回本大区。
- 查询产品排行和月度趋势时，只基于本大区订单数据。
- 如果显式询问其他大区，系统不会返回其他大区数据。

### 3. 销售总监：`SALES_DIRECTOR`

销售总监可以查看全公司数据。

具体表现：

- 不自动追加 `rep_id` 或 `region_id` 限制。
- 可以查询全公司订单、销售额、排行、产品表现、趋势和异常预警。
- 如果请求中指定大区或销售员，则按请求条件过滤。

## 六、异常预警类型

当前异常预警工具为 `detect_sales_anomalies`，会检测以下几类异常：

| 异常类型 | 含义 |
| --- | --- |
| 大区订单量骤降 | 最近 2 周订单量相较过去基线明显下降 |
| 产品连续零销售 | 活跃产品在可见范围内连续多天没有销售订单 |
| 销售员退单率异常 | 近 30 天销售员退单比例超过阈值 |
| 销售员业绩骤降 | 销售员近 30 天业绩较上一周期明显下降 |

## 七、不同角色可见的异常预警差异

### 1. 销售总监可见内容

`SALES_DIRECTOR` 可扫描全公司范围内的异常。

可见异常包括：

- 所有大区的订单量骤降预警。
- 全公司范围内产品连续零销售预警。
- 所有销售员的退单率异常预警。
- 所有销售员的业绩骤降预警。

参考场景中，总监可以看到：

- 华北区订单量骤降。
- 智能手表连续零销售。
- 王芳退单率高。
- 张磊业绩骤降。

总结：总监看到的是公司级完整风险视图。

### 2. 销售主管可见内容

`SALES_MANAGER` 只扫描自己所属大区的异常。

可见异常包括：

- 本大区的订单量骤降预警。
- 本大区范围内产品连续零销售预警。
- 本大区销售员的退单率异常预警。
- 本大区销售员的业绩骤降预警。

不可见内容：

- 其他大区的订单量骤降。
- 其他大区销售员的退单率异常。
- 其他大区销售员的业绩骤降。
- 只发生在其他大区销售范围内的产品零销售风险。

参考场景中，华东区主管只能看到华东区相关异常，例如王芳退单率高；看不到华北区骤降，因为华北区不属于其权限范围。

总结：主管看到的是本大区团队风险视图。

### 3. 普通销售员可见内容

`SALES_REP` 只扫描自己相关的异常。

当前实现中，普通销售员可见异常包括：

- 自己的退单率异常预警。
- 自己的业绩骤降预警。

不可见内容：

- 大区级订单量骤降预警。
- 产品连续零销售预警。
- 其他销售员退单率异常。
- 其他销售员业绩骤降。
- 其他大区或全公司级异常。

参考场景中，普通销售员王芳只能看到自己退单率偏高的提醒，看不到其他人的异常，也看不到大区级别预警。

总结：销售员看到的是个人风险视图，不暴露团队或公司级风险数据。

## 八、异常预警权限矩阵

| 异常预警类型 | `SALES_REP` 销售员 | `SALES_MANAGER` 主管 | `SALES_DIRECTOR` 总监 |
| --- | --- | --- | --- |
| 大区订单量骤降 | 不可见 | 仅本大区 | 全公司所有大区 |
| 产品连续零销售 | 不可见 | 仅本大区范围 | 全公司范围 |
| 销售员退单率异常 | 仅本人 | 仅本大区销售员 | 全公司销售员 |
| 销售员业绩骤降 | 仅本人 | 仅本大区销售员 | 全公司销售员 |

## 九、示例说明

假设当前数据中存在以下异常：

- 华北区订单量骤降。
- 智能手表连续零销售。
- 华东区销售员王芳退单率偏高。
- 华北区销售员张磊业绩骤降。

不同用户看到的结果如下：

| 登录用户 | 角色 | 可见异常 |
| --- | --- | --- |
| 销售总监 | `SALES_DIRECTOR` | 华北区骤降、智能手表零销售、王芳退单率高、张磊业绩骤降 |
| 华东区主管 | `SALES_MANAGER` | 王芳退单率高，以及其他华东区范围内异常 |
| 王芳 | `SALES_REP` | 自己退单率偏高 |
| 张伟 | `SALES_REP` | 只看张伟自己的退单率和业绩变化，不能看到王芳或张磊异常 |

## 十、当前实现与参考文献的对齐情况

已对齐：

- 三种角色的数据范围规则。
- Service 层作为权限过滤核心边界。
- 工具层异常预警按当前用户权限扫描可见范围。
- 普通销售员不能看到其他销售员和大区级异常。
- 销售主管不能看到其他大区异常。
- 销售总监可以看到全公司异常。

当前实现中的一个收敛点：

- 普通销售员不会看到“产品连续零销售”预警。该类预警更偏产品和区域运营风险，当前只开放给主管和总监。
- 普通销售员请求销售员排行或大区排行时，工具层会直接提示无权查看团队/大区排行，而不是返回只有本人数据的伪排行；普通销售员仍可查询个人产品销售排行和个人销售趋势。

## 十一、安全边界总结

权限过滤不是依赖模型自觉遵守，而是在代码层强制执行：

- JWT 确认当前用户身份。
- `CurrentUser` 保存当前用户角色和范围。
- `SalesQueryService` 对所有查询入口注入权限过滤。
- 异常检测工具通过 Service 获取可见范围，不直接扫描全量用户可见之外的数据。

因此，即使用户在对话中要求“查看全公司异常”或“查看其他大区异常”，只要其角色权限不足，系统也不会返回越权数据。
